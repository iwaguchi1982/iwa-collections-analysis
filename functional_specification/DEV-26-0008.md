# Auto Discovery の並列実行エンジン (DEV-26-0008)

## 1. 背景
全遺伝子または数万〜数十万の遺伝子ペアに対して Cox 比例ハザード解析や log-rank 検定を実行する「Auto Discovery」機能は、計算負荷が極めて高く、シングルプロセスでは数時間以上の時間を要する場合がある。

## 2. 目的
- マルチコア CPU を最大限に活用し、探索解析の全体時間を大幅に短縮する。
- サーバーの負荷状況に合わせて、使用する CPU コア数をユーザーが調整可能にする。

## 3. 基本設計方針
- `multiprocessing` モジュール（または joblib 等のライブラリ）を用いた、遺伝子単位の並列分散処理。
- 探索結果の中間の集約を行い、メモリ溢れを防止しつつ、最終的な上位 N 件（Top Results）を抽出する。

## 4. 機能詳細
### 4.1 Task Partitioning
- 全探索対象（Gene Pairs 等）を、コア数に応じた「チャンク」に分割。
- 各チャンク内での独立した生存時間解析（KM / Cox）の実行。

### 4.2 Resource Guardrails
- CPU 使用率の制限設定（n_jobs パラメータ）。
- 計算中のプログレス更新頻度の最適化（頻繁すぎる通知によるオーバーヘッドの防止）。

### 4.3 Fast Log-rank Implementation
- ネイティブな統計ライブラリを活用し、1解析あたりのオーバーヘッドを最小化する高速演算の実装。

## 5. UI構成
- **Compute Resource Slider (Advanced Settings)**: 使用コア数を 1 から最大数まで指定。
- **Real-time Progress Indicator**: 完了したタスク数と、完了予測時間（ETA）の表示。

## 6. 実装上の注意
- サブプロセス間での巨大なデータセット（Expression Matrix 等）の受け渡しがボトルネックにならないよう、共有メモリの活用や、適切なコピー抑止を検討する。
- エラーが発生した個別の解析（収束不全等）が、全体の計算プロセスを停止させないよう、強固な例外ハンドリングを子プロセス側に設ける。
- Streamlit の実行環境の再起動（Rerun）時に、進行中の並列プロセスが適切にクリーンアップされるように管理する。
