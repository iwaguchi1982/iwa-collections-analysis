# Iwa Collections Analysis: 🚀 GPUアクセラレーション導入に関する見積もりと展望

現在の `iwa-collections-analysis` アプリケーション（CPUベース実装）をプロファイリング・スキャンし、**GPU処理（CUDA / CuPy / PyTorch / RAPIDS cuML等）** を導入した場合に期待できるパフォーマンス向上（Speedup）と、その実装アプローチについて検討しました。

結論として、特定のボトルネック（特に **Auto Discovery** と **PSM**）において、**10倍〜100倍という劇的な高速化が可能**であり、現在の「数十秒〜数分待つ」体験を「ほぼリアルタイム（数秒以内）」に引き上げることができます。

---

## 1. 最大の恩恵: Auto Discovery (2-Gene 探索)
**予想される高速化: 約 50〜100倍 (1分 → 1秒未満)**

**▶ 現状のボトルネック (CPU実装)**
`src/auto_discovery.py` の実装を確認したところ、指定された `target_n`（例: 50〜500遺伝子）の全てのペアの組み合わせに対して **Pythonの二重ループ** を回し、その中で `lifelines` の `multivariate_logrank_test` を逐次実行しています。
- 100 遺伝子なら約 5,000 ペア
- 500 遺伝子なら約 125,000 ペア
CPUではループのオーバーヘッドと、1回ずつの行列スライス・ソート処理が極めて重く、数十分かかることもあります。

**▶ GPUアプローチ (PyTorch / CuPy)**
Log-rank検定の計算（各死亡時点での Observed vs Expected の差の集計）は、完全に**行列演算（Matrix Multiplication）**に落とし込むことができます。
データ（生存時間、生死、500個の遺伝子発現行列）を事前にGPUのVRAMに載せておき、一括でテンソル演算を行えば、125,000ペアのテスト統計量を「数発のGPUカーネル呼び出し」で同時に算出できます。

## 2. 傾向スコアマッチング (PSM)
**予想される高速化: 約 10〜30倍**

**▶ 現状のボトルネック (CPU実装)**
`ui/psm_view.py` における PSM は、Logistic Regression でスコアを出したあと、「Caliper制限付きの 1:1 Nearest Neighbor Greedy Matching」を走らせています。
患者数 (N) が1万人クラスのPanCancerデータセットになると、スコア間の距離行列（O(N^2)）の計算や、そのソート・最適なペアの検索がCPUのシングルスレッドで詰まります。

**▶ GPUアプローチ (RAPIDS cuML / PyTorch)**
- `cuML.LogisticRegression` で傾向スコアを瞬時に計算。
- 距離行列の計算をGPUのテンソルコアで行い、並列ソート (`cupy.argsort` や `torch.topk`) を用いてバッチでマッチング可能なペアを抽出します。
これにより、巨大なコホートでも数秒でマッチングと生存曲線の再描画が完了します。

## 3. その他の計算ヘビーな処理

### A. Differential Expression (DEG & PCA)
**予想される高速化: 約 5〜10倍**
- **現状**: `scipy.stats.ttest_ind` を数万遺伝子に対して適用。現在のCPU（Numpy/Scipyベクタライズ）でも数秒で終わります。
- **GPU化**: `cupy` の配列にデータを載せ替えて平均・分散・T値を一括計算すれば、ほぼ一瞬（0.1秒以下）でVolcano Plotの描画まで到達できます。
- PCAについても、`cuML.PCA` を使えばCPUの1/10の時間で分解が完了します。

### B. Auto Cutoff Smoothness Scan
**予想される高速化: 約 10倍**
- **現状**: 20〜80 Percentileの全60分割点に対して、順番にCox回帰を走らせています。
- **GPU化**: Coxの部分尤度（Partial Likelihood）の最適化は、Newton法を直列に回すため完全並列化はやや難しいですが、PyTorch等で「バッチ処理として60個のモデルを同時に勾配降下させる（例: `pycox` の活用）」アーキテクチャに変更すれば、一気に高速化できます。

## 💻 導入にあたってのアーキテクチャ変更案

現在、すべてのインフラ要件を「ピュアPython（CPU）」に収めているため、環境構築が非常に手軽であるという大きなメリットがあります（PixiやDockerで一発起動）。
もし GPU Mode を導入する場合、以下のような戦略が推奨されます：

1. **オプショナルな依存関係 (`[gpu]`)**
   - デフォルトは今のままCPUで動くようにし、`torch` や `cupy`, `cuml` は必須環境にしない。
   - `if has_gpu_env:` のような分岐を作り、NVIDIAドライバとライブラリがあるサーバー内でのみ「GPUアクセラレーションモード」のトグルが有効化されるようにする。

2. **JITコンパイラ (Numba) の活用**
   - 大掛かりなGPUライブラリを入れずとも、手始めに Auto Discovery の二重ループ部分に `numba` （CPUのJITコンパイル）や `numba.cuda` とマルチスレッド (`prange`) を当てるだけでも、**CPUのままで 5〜10倍 クラスの現実的な高速化**が見込めます。まずはここから着手するのが最もコスパが高いアプローチです。
